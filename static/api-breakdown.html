What is an API?
Application Programming Interface — the waiter between your browser and the server
Restaurant analogy:

You (browser) sit at a table. The kitchen (server + agent) has all the food (data + AI).
You can't walk into the kitchen. You give your order to the waiter (API).
The waiter goes to the kitchen, gets the food, and brings it back to you.

The waiter speaks a specific language — you can't just shout random things. You must use the menu (endpoints) with specific formats.
Without API
Browser would have to directly access Python code, databases, files. Impossible. Insecure.
With API (FastAPI)
Browser sends a clean HTTP request to a URL. FastAPI receives it, processes it, sends back a response. Clean and safe.
In Boozo.ai: FastAPI is the API layer. Your browser talks to localhost:8080. Everything goes through FastAPI — loading the page, Google OAuth login, session management, sending messages, getting responses.

GET vs POST
Two different ways to send a request
Simple rule:
GET = "give me something" — just fetching, no data sent
POST = "here is something, process it" — sending data to the server
GETFetching
Like reading a menu — you ask to see something, nothing changes on the server.

In Boozo.ai:
GET / → get the HTML page
GET /health → check if server is alive

Data goes in the URL. Visible in browser bar.
POSTSending data
Like placing an order — you send data to the server and it does something with it.

In Boozo.ai:
POST /chat → send your message + history + image to the agent

Data goes in the request body as JSON. Hidden from URL bar.
// What GET /health looks like over the network: GET /health HTTP/1.1 Host: localhost:8000 // Response: HTTP/1.1 200 OK {"status": "ok"} ───────────────────────────────────────── // What POST /chat looks like over the network: POST /chat HTTP/1.1 Host: localhost:8000 Content-Type: application/json { "message": "what is Python?", "history": [], "selected_model": "groq:llama-3.3-70b-versatile" }

Pydantic — Data Bodyguard
Validates every incoming request automatically
What it does: When a POST request arrives at /chat, the raw JSON body gets handed to Pydantic's ChatRequest model. Pydantic checks every field, converts types, fills defaults — then gives your code a clean Python object. If anything is wrong, it auto-rejects with a clear error.
class ChatRequest(BaseModel): message: str # REQUIRED — must be text history: list[dict] = [] # optional — list of past turns image: str | None = None # optional — base64 image image_mime: str | None = None # optional — "image/png" etc selected_model: str | None = None # optional — which LLM to use
Valid request
{
  "message": "hello",
  "history": [],
  "selected_model": "groq:llama-3.3-70b"
}

→ ChatRequest object created
→ chat() function called
Invalid request
{
  "history": "not a list"
}
(message missing!)

→ 422 Validation Error
→ chat() never called
After Pydantic, your code accesses the data cleanly:

req.message → "what is Python?"
req.history → []
req.selected_model → "groq:llama-3.3-70b-versatile"
req.image → None (no image sent)

How Tokens Are Generated
The core mechanic of how every LLM produces text
The single most important idea in all of AI:

An LLM does ONE thing — it looks at all the text so far and predicts the next token. Then it adds that token and predicts the next one. And again. And again. That's how "Hello, how are you?" is generated — one word-chunk at a time.
Step by step — generating "Python is great":
📥
Input prompt sent to LLM
"You are Boozo.ai... User: tell me about Python"
probability over vocab
🎲
LLM assigns probability to every possible next token
"Python" = 42%   "Programming" = 18%   "The" = 9%   "I" = 7% ...
temperature 0.3 applied
🌡️
Temperature sharpens the probabilities
At 0.3: most likely token wins almost always → "Python" selected
append + repeat
➕
"Python" added. Now predict next token...
New input: "...User: tell me about Python\nAssistant: Python"
repeat until done token
🏁
Stops at special [END] token or max_tokens limit
max_tokens = 1024 in Boozo.ai — prevents infinite generation
Watch it happen — live token build:
Why this matters for Boozo.ai: The LLM doesn't "think" and then "write". It generates one token, adds it, generates another. This is why temperature matters — it controls how "adventurous" each token pick is. At 0.3, picks are focused and accurate. At 1.0, picks are random and creative.
Token count in Boozo.ai: Each iteration, agent.py estimates token usage with len(prompt) // 4 (roughly 4 characters per token). This is shown in the context window bar. The LLM can process max NUM_CTX = 8192 tokens at once. If exceeded, the model starts forgetting earlier parts of the conversation.

SSE — Server-Sent Events
How events flow from server to browser in real time
Normal HTTP: browser asks → waits → gets full response at once.
SSE: browser asks → connection stays open → server sends pieces one by one → browser updates live.

This is why you see Thought, Action, Observation appear one by one instead of all at the end.

New in v3 — True Groq streaming: When Groq is selected, call_groq_stream() opens Groq's own SSE stream. The agent detects "Final Answer:" in the incoming token buffer and immediately starts forwarding tokens as answer_token events — the answer appears word by word as the LLM generates it, not after it finishes.
Exact SSE format — what travels over the wire:
# server side (agent.py) — yielding events yield {"type": "iteration", "n": 1, "max": 5} yield {"type": "tokens", "used": 312, "limit": 8192} yield {"type": "thought", "content": "I need to search..."} yield {"type": "action", "content": "web_search(Python version)"} yield {"type": "observation", "content": "Python 3.13..."} yield {"type": "answer_token", "content": "P"} yield {"type": "answer_token", "content": "y"} yield {"type": "done", "content": "Python 3.13 is the latest..."} # main.py — wraps each event in SSE format yield f"data: {json.dumps(event)}\n\n"
// browser side (index.html) — reading SSE const resp = await fetch('/chat', { method: 'POST', body: ... }); const reader = resp.body.getReader(); while(true) { const {value, done} = await reader.read(); // parse "data: {...}\n\n" const ev = JSON.parse(line.slice(6)); if (ev.type === 'thought') addBrainStep('thought', ev.content); if (ev.type === 'action') addBrainStep('action', ev.content); if (ev.type === 'answer_token') aMsg.textContent += ev.content; if (ev.type === 'chart') renderChart(ev.content); if (ev.type === 'done') finalise(ev.content); }
All SSE event types
iteration — loop number
tokens — context usage
timing — ms per step
model_used — which LLM
thought — agent thinking
action — tool being called
observation — tool result
scratchpad — full rough work
answer_token — one character
chart — Chart.js spec
done — final answer
error — something failed
Why not WebSockets?
WebSockets are two-way (browser AND server can send). SSE is one-way (server → browser only). For a chat app where the browser just sends one POST and then listens, SSE is simpler and perfect. WebSockets would be overkill.
