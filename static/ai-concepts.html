Concept 01
🧠
LLM — Large Language Model
The giant AI brain that answers questions
Used in: agent.py — call_groq, call_cerebras, call_gemini, call_ollama
Simple analogy Imagine a student who read every book, website, and article ever written. You ask them a question and they answer from memory. That student is an LLM. Groq, Gemini, Ollama, Cerebras — all LLMs, just different students.
An LLM is a neural network trained on billions of text examples. During training it learns one simple skill: predict the next word. But do that millions of times at high quality and it starts to seem like it "understands" language. It doesn't think — it predicts patterns.

In Boozo.ai, the LLM is the core engine. Every question you ask goes to an LLM. The LLM reads the prompt and produces the next words — which could be a Thought, an Action, or a Final Answer.
Models in Boozo.ai
Groq: llama-3.3-70b
Cerebras: llama-3.3-70b
Gemini: gemini-2.0-flash
Ollama: llama3.2 (local)
What 70b means
70 billion parameters — like 70 billion tiny knobs tuned during training. More knobs = smarter model. The llama-3.3-70b is very capable.
Concept 02
🔤
Tokens — How LLMs Count Text
LLMs don't read letters or words — they read chunks
Used in: agent.py — NUM_CTX = 8192, token estimate (len(prompt) // 4)
Simple analogy Instead of reading letter by letter, LLMs break text into small chunks called tokens. Think of tokens like Lego bricks. "Python" might be 1 token. "programming" might be 2. A space + "is" = 1 token. On average, 1 token ≈ 4 characters or about 0.75 words.
The sentence "Hello, how are you?" broken into tokens:
Hello , how are you ?
Why tokens matter: LLMs have a maximum number of tokens they can process at once (the context window). Boozo.ai estimates token usage with len(prompt) // 4 and shows it in the right panel bar so you can see how full the context window is.
Concept 03
✍️
Prompt Engineering
The art of writing instructions that make the LLM behave correctly
Used in: prompts.py — SYSTEM_PROMPT, REACT_TEMPLATE
Simple analogy An LLM is like a very smart but obedient genie. If you say "give me something to eat" it might give you mud. If you say "give me a hot cheese pizza with extra toppings" it gives you exactly that. Prompt engineering = writing clear, specific instructions so the genie does exactly what you want.
Boozo.ai uses two prompts:

1. System Prompt (prompts.py) — tells the LLM WHO it is, WHAT tools it has, and HOW to respond. It's like the employee handbook.

2. ReAct Template — the actual message sent each iteration. It includes the system prompt + conversation history + user question + scratchpad. It's the full briefing the LLM gets before answering.

Concept 04
🔄
ReAct — Reason + Act
The pattern that turns a chatbot into an agent
Used in: agent.py — run_agent() loop, MAX_ITERATIONS = 5
Simple analogy Normal chatbot: You ask → it answers. Done.
ReAct agent: You ask → it thinks → it acts (uses a tool) → it thinks again → uses another tool → thinks → answers. Like a detective who investigates before concluding.
ReAct was invented by researchers at Google in 2022. The key insight: if you ask an LLM to write its reasoning step by step AND let it call external tools, it becomes dramatically more accurate than just answering directly.

In Boozo.ai, the loop runs up to 5 iterations. Each iteration: build prompt → call LLM → parse output → if tool needed run it → repeat. When the LLM writes Final Answer: the loop stops and the answer streams to you.
Without ReAct
LLM answers from memory only. Can hallucinate (make up facts). Can't search, can't calculate, can't draw charts.
With ReAct (Boozo.ai)
LLM plans, searches the web, runs code, checks results, then gives a verified answer. Much more accurate.
Concept 05
📝
Scratchpad — The Agent's Rough Work
Fake memory between LLM calls
Used in: agent.py — scratchpad string, grows each iteration
Simple analogy Every time you call the LLM it forgets everything. It's like waking up with amnesia every 5 seconds. The scratchpad is the note you paste in front of it saying "here's what you already did." Without it, the agent would repeat the same step forever.
Here is what a real scratchpad looks like after 2 iterations:
The scratchpad starts empty (scratchpad = "") and grows after each step. It gets pasted into the prompt every time. This is the core trick that gives the agent "memory" across multiple LLM calls. You can watch it grow live in Boozo.ai's right panel.
Concept 06
🛠️
Tool Use — Agent's Hands
How the agent interacts with the real world
Used in: tools.py — web_search, code_exec, calculator, rag_ingest, rag_search, plot
Simple analogy The LLM is a brain in a jar — brilliant but can't touch anything. Tools are like robot arms attached to the jar. The brain says "search for X" → the arm goes and does it → brings back the result. The brain now knows things it couldn't have known from training alone.
The LLM writes Action: web_search(latest Python version) in its output.
Your code (dispatch_tool in tools.py) reads this, finds the web_search function, runs it, and pastes the result back as an Observation in the scratchpad.

Boozo.ai's 6 tools:
web_search
Scrapes DuckDuckGo HTML — no API key. Returns top 4 result snippets.
code_exec
Runs Python in a sandbox. AST-validated. Returns stdout output.
calculator
Safe maths only. Runs in subprocess with 2s OS kill timeout.
rag_ingest
Stores URLs or text into ChromaDB vector database.
rag_search
Semantic search over stored documents. Returns top 3 chunks.
plot
Takes JSON chart spec → renders interactive Chart.js chart in chat.

Context Window — LLM's Short-Term Memory
The maximum amount of text the LLM can see at once
Used in: agent.py — NUM_CTX = 8192, tokens event in SSE stream
Simple analogy Imagine you can only hold 10 pages of notes in front of you at once. If your conversation grows to 11 pages, the first page falls off the back. The LLM literally cannot see text beyond its context window — it's like those pages don't exist.

Concept 08
🌡️
Temperature — Creativity Knob
Controls how random or focused the LLM's answers are
Used in: agent.py — "temperature": 0.3 in every LLM call
Simple analogy Think of temperature like a creativity dial on a robot writer. Turn it to 0 — robot writes the most boring, predictable sentences possible. Turn it to 1 — robot writes wild, creative, sometimes nonsense poetry. For an AI assistant, you want it focused but not robotic.
0.0
Robotic
Always picks the single most likely next word. Repetitive and safe.
0.3
Boozo.ai uses this
Mostly focused but allows some natural variation. Best for technical answers.
0.7
Creative
Good for writing stories, brainstorming, poetry. Less precise.
1.0+
Wild
Random, surprising, often incoherent. Avoid for factual tasks.
Concept 09
📚
RAG — Retrieval Augmented Generation
Give the LLM a personal, searchable notebook
Used in: tools.py — rag_ingest, rag_search, ChromaDB, sentence-transformers
Simple analogy The LLM only knows what it learned during training (which ended months ago). RAG is like giving it a personal notebook. You write stuff in the notebook. When you ask a question, the LLM checks the notebook first, then answers. Now it can know things that weren't in its training data.
How it works in Boozo.ai:

Step 1 — Ingest: You tell the agent to "remember" a URL or text. It scrapes the content, splits it into 300-character chunks, converts each chunk into a number vector (embedding), and stores everything in ChromaDB.

Step 2 — Retrieve: You ask a question. The agent converts your question into a vector too, then finds the stored chunks whose vectors are closest in meaning. Top 3 chunks are given to the LLM as extra context.

Step 3 — Generate: The LLM now has real, specific information to answer from — not just training data guesses.

Concept 10
🔢
Vector Embeddings
Turning words into numbers so computers understand meaning
Used in: tools.py — SentenceTransformerEmbeddingFunction, all-MiniLM-L6-v2
Simple analogy Imagine plotting words on a map. "Dog" and "puppy" would be placed very close together. "Dog" and "aeroplane" would be far apart. An embedding model creates this map — converting each word/sentence into a list of numbers that represents where it sits on the meaning map. Closer numbers = similar meaning.
The model used: all-MiniLM-L6-v2 (80MB, runs on your CPU). It converts any sentence into a list of 384 numbers called a vector. When you search, your query also becomes 384 numbers. ChromaDB then measures cosine similarity (the angle between two number lists) to find the closest match.

Why this matters: You can search for "programming language" and find documents about "Python coding" even if those exact words don't appear — because their meaning vectors are close.
"Python is fun"
[0.23, -0.11, 0.87, 0.04, ...]
384 numbers total
"I love coding"
[0.21, -0.09, 0.84, 0.06, ...]
Very similar numbers = similar meaning
Concept 11
📡
Token Streaming — Letter by Letter Output
Why text appears gradually instead of all at once
Used in: agent.py — answer_token loop, main.py — StreamingResponse + SSE
Simple analogy Old way: order a pizza, wait 40 minutes, it arrives all at once. Streaming: a chef cooks in front of you and hands you each slice as it comes out of the oven. You start eating at minute 5 instead of minute 40. Streaming feels faster even if total time is the same.
Boozo.ai streams at two levels:

1. SSE stream (main.py) — FastAPI's StreamingResponse sends each agent event (thought, action, observation) immediately as a data: {...}\n\n message.

2. Token stream (agent.py) — When the final answer is ready, it loops through each character and yields an answer_token event: for char in answer: yield type: "answer_token", content: char

The browser's JavaScript reads the SSE stream and appends each character to the chat bubble in real time.

Fallback Chain — Always Stay Online
Automatic backup providers when the first one fails
Used in: agent.py — call_llm() with try/except chain
Simple analogy You call your first taxi. No answer. You call the second. Busy. You call the third. It comes. You always get to your destination — the user never sees the failed calls. That's a fallback chain. Boozo.ai has 4 taxis.
Try Groq
fail →
Try Cerebras
fail →
Try Gemini
fail →
Ollama (local)
→
Always get answer
Each try is wrapped in a try/except block. If Groq throws any exception (rate limit, timeout, server error) the except catches it silently and tries Cerebras. The user sees which provider was actually used via the model badge in the header.
Cloud providers (Groq, Cerebras, Gemini)
Fast. Free tier. Run on NVIDIA H100 clusters. Latency: 0.5–3 seconds.
Ollama (local)
Runs on your own machine. Completely private. No internet needed. Slower (15–60s) but always available.
