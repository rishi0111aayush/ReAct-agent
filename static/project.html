How SSE Streaming Works

Normal HTTP: you ask, server replies, connection closes. SSE (Server-Sent Events): connection stays open, server sends data in chunks as they become available.

// Frontend JavaScript — simplified
async function sendMessage() {
  const response = await fetch('/chat', {
    method: 'POST',
    body: JSON.stringify({ message: "What is Python?" })
  });

  const reader = response.body.getReader();  // open streaming reader
  let answer = '';

  while (true) {
    const { value, done } = await reader.read();
    if (done) break;

    const text = new TextDecoder().decode(value);
    const event = JSON.parse(text.replace('data: ', ''));

    if (event.type === 'answer_token') {
      answer += event.content;        // accumulate tokens
      showInChat(answer);             // render incrementally
    }
    if (event.type === 'done') break;
  }
}
Key variable: window._chatHistory — stores the conversation history array. Uses window._chatHistory (not let history) because multiple <script> blocks in HTML each have their own scope. Using window. makes it truly global across all script blocks.

The Chat Endpoint — Step by Step

@app.post("/chat")
def chat(req: ChatRequest, request: Request):

    # Step 1: Validate the message
    if len(req.message.strip()) == 0:
        raise HTTPException(status_code=400, detail="Message cannot be empty.")
    if len(req.message) > 4000:
        raise HTTPException(status_code=400, detail="Message too long.")

    # Step 2: Check rate limit (30 requests/minute per user)
    user   = current_user(request)
    rl_key = user["id"] if user else request.client.host
    allowed, count = cache.check_rate_limit(rl_key)
    if not allowed:
        raise HTTPException(status_code=429, detail=f"Rate limit exceeded ({count}/min)")

    # Step 3: Stream the agent response
    def event_stream():
        for event in run_agent(req.message, req.history, ...):
            yield f"data: {json.dumps(event)}\n\n"

    return StreamingResponse(event_stream(), media_type="text/event-stream")

Google OAuth — How Login Works

  • 1
    User clicks "Sign in with Google"

    Browser navigates to GET /auth/google. Server redirects to accounts.google.com with your app's client_id.

  • 2
    User approves on Google's page

    Google authenticates the user and redirects back to GET /auth/callback?code=xyz.

  • 3
    Server exchanges the code for user info

    authlib calls Google's API using the code. Gets back: email, name, picture, sub (Google's unique user ID).

  • 4
    User saved to SQLite, cookie set

    db.upsert_user() creates or updates the user record. request.session["user_id"] = google_id sets an encrypted cookie valid for 30 days.

  • 5
    Every future request is identified

    The browser sends the cookie automatically. current_user(request) reads request.session["user_id"] and looks up the DB.

Rate Limiting: Without Redis — every user can send max 30 messages per minute. If they exceed this, they get HTTP 429. Logged-in users are tracked by their Google ID. Anonymous users are tracked by their IP address.
Section 04

Agent — agent.py (The Brain)

This is the most important file. It implements the ReAct (Reason + Act) pattern — a loop where the AI thinks, uses a tool, observes the result, thinks again, and repeats until it has a final answer.

What is ReAct?

ReAct stands for Reason and Act. It is a technique where instead of having the AI answer immediately, you force it to show its thinking step by step:

# Example: User asks "What is the weather in Paris and plot it?"

Thought: I need to get the weather for Paris first.
Action: get_weather(Paris)
Observation: Paris: Partly cloudy +18°C, Wind: 12 km/h, Humidity: 65%

Thought: I have the data. Now I should plot it as a chart.
Action: plot({"type":"bar","title":"Paris Weather","labels":["Temp°C","Wind km/h","Humidity%"],"datasets":[{"label":"Value","data":[18,12,65]}]})
Observation: CHART:{...json...}

Thought: I have both the weather data and the chart rendered.
Final Answer: Here is the current weather in Paris...

The ReAct Loop in Code

def run_agent(user_message, history, image, image_mime, selected_model, user_id):
    for iteration in range(5):    # max 5 iterations

        # Build the full prompt (system + history + user + scratchpad)
        prompt = REACT_TEMPLATE.format(...)

        # Ask the LLM
        llm_output = call_groq_stream(prompt)   # streams tokens in real time

        # Parse what the LLM said
        parsed = parse_llm_output(llm_output)
        # parsed = { thought: "...", action: "get_weather(Paris)", final_answer: None }

        if parsed["final_answer"]:
            yield {"type": "done", "content": parsed["final_answer"]}
            return                      # stop the loop

        if parsed["action"]:
            # Run the tool (e.g. get_weather("Paris"))
            observation = dispatch_tool(parsed["action"])
            scratchpad += f"Observation: {observation}\n\n"
            # Loop again — LLM sees the observation next iteration

LLM Providers — Fallback Chain

The agent tries LLM providers in order. If one fails (quota exceeded, server error), it silently moves to the next:

FALLBACK 1

FALLBACK 2

FALLBACK 3

Ollama — local llama3.2

Runs on your own machine. Works offline. Slower but zero cost, zero data sharing.

Token Streaming Explained

When using Groq, the response is streamed token by token (a token is roughly 4 characters). This is how the text appears to "type itself" in the UI:

def call_groq_stream(prompt):
    payload = {"stream": True, ...}    # tell Groq to stream

    with httpx.stream("POST", GROQ_URL, json=payload) as resp:
        for line in resp.iter_lines():
            # Each line looks like:  data: {"choices":[{"delta":{"content":"The"}}]}
            token = parse(line)["delta"]["content"]
            yield token   # "The" → "ure" → "is" → " a" → " concept..."
Scratchpad: As the loop runs, a scratchpad string accumulates all Thoughts, Actions, and Observations. This is appended to the next prompt so the LLM "remembers" what it already tried. After 5 iterations, the agent gives up and returns an error.

parse_llm_output() — Reading the LLM's Mind

The LLM returns plain text. This function uses regex to extract the three possible parts:

def parse_llm_output(text):
    # Regex to find "Thought: ..." up until "Action:" or "Final Answer:"
    thought_match = re.search(r'Thought:\s*(.+?)(?=Action:|Final Answer:|$)', text, re.DOTALL)
    action_match  = re.search(r'Action:\s*(.+?)(?=Thought:|Observation:|Final Answer:|$)', text, re.DOTALL)
    answer_match  = re.search(r'Final Answer:\s*(.+)', text, re.DOTALL)

    return {
        "thought":      thought_match.group(1).strip() if thought_match else None,
        "action":       action_match.group(1).strip()  if action_match  else None,
        "final_answer": answer_match.group(1).strip()  if answer_match  else None,
    }
Section 05

Prompts — prompts.py

Prompts are the instructions you give the AI to control how it behaves. Without a good prompt, the AI would not know it should use tools, or how to format its response.

SYSTEM_PROMPT — The AI's Job Description

This is sent to the LLM at the start of every request. It defines:

  • Identity: "You are Boozo.ai, an expert AI assistant..."
  • Tool list: Every tool's name, signature, and when to use it
  • Response format: Must use Thought → Action → Final Answer structure
  • Rules: Always start with Thought, never skip it, etc.
Why does format matter? If the prompt doesn't specify the exact format, the LLM might write I will search for this: instead of Action: web_search(...). The regex parser (parse_llm_output) would then fail to find the action. A strict format in the prompt = reliable parsing.

REACT_TEMPLATE — The Full Prompt Builder

# prompts.py
REACT_TEMPLATE = """{system}

{history}

Section 06

Tools — tools.py (The Agent's Hands)

Tools are functions the AI can call to interact with the real world — search the internet, run code, check the weather, remember things. Without tools, the AI can only use its training data.

web_search(query)
Searches DuckDuckGo and returns top 4 results. Results are cached in Redis for 1 hour — if you ask the same thing twice, second request is instant.
web_search("Python 3.13 new features") → "- What's new in 3.13: Python 3.13 ships..."
code_exec(code)
Runs Python code in a sandbox. Only safe built-ins allowed (print, range, len, math). No file access, no imports, no attribute access. Output captured from stdout.
code_exec("print(sum([1,2,3,4,5]))") → "15"
calculator(expr)
Evaluates math expressions in a completely isolated subprocess — even safer than code_exec. Regex validates only math operators allowed first.
calculator("(15 ** 2 + 144) ** 0.5") → "17.69"
rag_ingest(source)
Ingests a URL or raw text into ChromaDB (vector database). Splits text into 300-char chunks with 50-char overlap. URL content is Redis-cached for 24h.
rag_ingest("https://docs.python.org/3/") → "Ingested 142 chunks from docs..."
rag_search(query)
Semantic search over the ChromaDB knowledge base. Uses sentence embeddings to find chunks similar in meaning, not just keyword matches.
rag_search("how does garbage collection work") → "[source: docs.python.org] CPython uses..."
plot(json_spec)
Renders an interactive Chart.js chart in the chat. Returns a special CHART:{json} string — the frontend detects this prefix and renders the chart.
plot({"type":"pie","title":"Languages","labels":["Python","JS"],"datasets":[{"data":[60,40]}]}) → CHART:{"type":"pie",...}
remember(fact)
Stores a fact in per-user Redis memory. Survives server restarts and new chat sessions. Requires login (uses Google ID as key).
remember("User prefers dark mode") → "Remembered: User prefers dark mode"
recall(topic)
Retrieves stored facts relevant to a topic. Filters by keyword match, falls back to last 10 facts if no matches. Requires login.
recall("preferences") → "Memories:\n- User prefers dark mode\n- User uses Python"
datetime_now(tz)
Returns current date, time, and weekday in any timezone. AI always uses this before answering "what day is it?" questions.
datetime_now("Asia/Kolkata") → "Friday, 04 July 2025 — 14:32:17 Asia/Kolkata"
wikipedia_search(query)
Fetches Wikipedia's REST API summary for a topic. Gets the opening paragraph — fast, reliable, no scraping needed.
wikipedia_search("Dijkstra's algorithm") → "**Dijkstra's algorithm** is an algorithm for finding the shortest paths..."
get_weather(city)
Uses wttr.in (free, no API key). Returns temperature, conditions, wind, humidity in a single line. Works for any city in the world.
get_weather("Tokyo") → "Tokyo: Sunny +28°C ↑ Wind: 8km/h Humidity: 70%"

How dispatch_tool() Works

When the LLM writes Action: web_search("AI news"), this function parses that string and calls the right function:

def dispatch_tool(action_str: str) -> str:
    # Input: 'web_search("AI news")'
    match = re.match(r"(\w+)\((.+)\)", action_str.strip(), re.DOTALL)

    tool_name = match.group(1)   # "web_search"
    arg       = match.group(2)   # '"AI news"'  → strips quotes → "AI news"

    tool = TOOLS.get(tool_name)  # looks up the Python function
    return tool(arg)              # calls web_search("AI news")

Code Sandbox Security

The code_exec tool uses two layers of protection:

Layer 1 — AST Analysis: Before running any code, it parses the code into an Abstract Syntax Tree. If it finds any attribute access (like os.system, open(), __import__), it rejects the code entirely.
Layer 2 — Restricted globals: Only whitelisted built-ins are available — print, range, len, int, float, str, list, dict, sum, max, min, abs, round, sorted, enumerate, zip, map, filter, isinstance. No import, no file system, no network.
Section 07

Cache — cache.py (Redis)

Redis is an in-memory database used for speed and temporary storage. In Boozo.ai it serves four different purposes.

🔍
CACHE — 1 HOUR TTL

Search Cache

When the agent searches DuckDuckGo, the result is stored with key boozo:search:{sha256hash}. Next time the same query is asked, the answer is returned instantly from Redis — no HTTP request made.

🌐
CACHE — 24 HOUR TTL

URL/Page Cache

When RAG ingests a URL, the scraped text is stored with key boozo:url:{sha256hash}. If the same URL is ingested again within 24 hours, the cached text is used instead of re-fetching.

🧠
MEMORY — PERMANENT

User Memory

When you say "remember X", the fact is RPUSH-ed onto a Redis list at boozo:mem:{user_id}. No TTL — it lives forever. recall(topic) does LRANGE 0 -1 to get all stored facts.

🛡️
SECURITY — 60 SEC WINDOW

Rate Limiting

Uses INCR + EXPIRE atomically via pipeline. Each request increments a counter. If counter exceeds 30 in 60 seconds, the request is blocked with HTTP 429.

Rate Limiting — How INCR + EXPIRE Works

def check_rate_limit(identifier: str):
    key  = f"boozo:rl:{identifier}"    # e.g. "boozo:rl:192.168.1.1"
    pipe = redis.pipeline()             # atomic batch
    pipe.incr(key)                      # increment counter (creates if not exists)
    pipe.expire(key, 60)               # reset after 60 seconds
    count = pipe.execute()[0]          # [count_result, expire_result][0]
    return count <= 30, count          # (allowed, current_count)

# First request:  count=1  → allowed (1 ≤ 30)
# 30th request:   count=30 → allowed (30 ≤ 30)
# 31st request:   count=31 → BLOCKED (31 > 30)
# After 60 secs:  key expires → count resets to 0

Graceful Fallback

If Redis is not running (e.g., local dev without Redis installed), every Redis operation silently becomes a no-op:

def _client(self):
    try:
        r = redis.from_url(REDIS_URL, socket_connect_timeout=2)
        r.ping()              # test connection
        self._r = r
    except Exception:
        self._r = None       # Redis unavailable — just don't cache
    return self._r

# Result: App works normally, just without caching and rate limiting
# rate limit check returns (True, 0) — everyone is allowed

Database — db.py (SQLite)

SQLite is a file-based SQL database. All users, sessions, and messages are stored permanently here. Unlike Redis, this data survives forever.

The Three Tables

-- users: one row per Google account
CREATE TABLE users (

    updated_at TEXT,

-- messages: every single chat message
CREATE TABLE messages (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL,
    role       TEXT CHECK(role IN ('user', 'assistant')),
    content    TEXT NOT NULL,
    created_at TEXT,
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE

When you start a new session, it's called "New Chat". After you send the first message, the session is automatically renamed to the first 52 characters of that message:

Ownership Verification

Every session operation checks that the session belongs to the requesting user. You cannot read, rename, or delete another user's sessions:

def get_messages(session_id, user_id):
    # First: verify ownership
    owns = db.execute(
        "SELECT 1 FROM sessions WHERE id=? AND user_id=?",
        (session_id, user_id)
    ).fetchone()

    if not owns:
        return None   # → HTTP 404 (don't reveal it exists)

    # Then: return messages
    return db.execute("SELECT * FROM messages WHERE session_id=?", ...)

ON DELETE CASCADE

Cascade means: If a user is deleted → all their sessions are deleted automatically. If a session is deleted → all its messages are deleted automatically. You never have orphaned messages with no parent session.

Section 10

Markdown Rendering Pipeline

The AI's raw text output is converted to beautiful formatted HTML in 4 steps.

  • 1
    Agent outputs raw Markdown text

    The LLM naturally writes formatted text: headings with ##, bold with **, code blocks with backticks, bullet lists with -, tables with pipes |.

  • 2
    marked.js converts Markdown → HTML

    marked.parse(text) transforms the raw string. **bold** becomes <strong>bold</strong>. A code block becomes <pre><code class="language-python">...</code></pre>.

  • 3
    DOMPurify strips dangerous HTML

    DOMPurify.sanitize(html) removes any <script>, onerror=, javascript: or other XSS vectors. Safe tags like <strong>, <code>, <table> are kept.

  • 4
    highlight.js + addCopyButtons() finish the job

    highlight.js scans for <code> blocks and adds color-coded syntax classes (keywords, strings, etc.). addCopyButtons() adds a "Copy" button to each code block.

```python
def binary_search(arr, target):
    lo, hi = 0, len(arr)-1
    while lo <= hi:
        mid = (lo+hi)//2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            lo = mid+1
        else:
            hi = mid-1
    return -1
```

Section 11

RAG — ChromaDB Vector Search

RAG stands for Retrieval-Augmented Generation. It lets the AI search through documents you've given it — beyond its training data.

How a Vector Database Works

Normal databases search by exact keywords. ChromaDB uses embeddings — every piece of text is converted to a list of ~384 numbers (a vector) that represents its meaning. Similar meanings = similar vectors.

Example: "How does memory management work in Python?" and "Python garbage collection explained" have very different words, but their meaning-vectors are close together. rag_search finds both even if your ingested document used different words than your query.
Model used: all-MiniLM-L6-v2 from sentence-transformers. It converts any text to a 384-dimensional vector. It runs locally — no API key needed, no data sent externally.

Chunking — Why We Split Text

# A 10,000-word article is split into manageable pieces
chunk_size = 300   # characters per chunk
overlap    = 50    # characters shared between consecutive chunks

# Text: "Python is a programming language. It was created by..."
# Chunk 0: "Python is a programming language. It was created by Guido van..."
# Chunk 1: "...Guido van Rossum in 1991. Python emphasizes readability..."
# Chunk 2: "...readability and uses indentation to define code blocks..."

# Overlap prevents important context from being split at boundaries

Persistent Storage

ChromaDB uses PersistentClient — data is saved to disk in the DATA_DIR/chroma/ directory. If the server restarts, all ingested documents are still there. No re-ingesting needed.

Lazy loading: ChromaDB (and its sentence-transformer model) is only loaded on the first actual use — not on server startup. This prevents a 30-second hang when the server starts, because downloading the embedding model takes time.

XSS (Cross-Site Scripting) Frontend — index.html DOMPurify.sanitize() strips all script tags before injecting AI response into DOM
Code injection via code_exec tools.py — code_exec() AST check rejects attribute access; restricted globals whitelist prevents imports
Math expression injection via calculator tools.py — calculator() Regex allows only 0-9 + - * / ( ) . % ** before evaluation; runs in isolated subprocess
Rate abuse / DDoS main.py + cache.py 30 requests/minute per user (by Google ID) or IP. Redis INCR+EXPIRE atomic counter.
Empty or oversized messages main.py HTTP 400 if empty or > 4000 characters
Accessing another user's sessions db.py — all session functions Every query includes AND user_id = ?. Returns None (→ 404) if no match.
Session hijacking main.py — SessionMiddleware Cookie is HMAC-signed with SECRET_KEY. Tampered cookies are rejected.
SQL injection db.py All queries use parameterized statements (?, ?, ?) — never string concatenation.
Agent prompt injection prompts.py + agent.py MAX_ITERATIONS=5 prevents infinite loops. Strict output format makes injection harder to hide.

The agent sends structured JSON events over the SSE stream. The frontend handles each type differently.

Event typeExample payloadWhat the frontend does
iteration{n: 1, max: 5}Shows "Thinking... (step 1/5)" indicator
tokens{used: 512, limit: 8192}Shows token usage bar
thought{content: "I need to search..."}Shows collapsible "Thought" bubble in chat
action{content: "web_search(Python)"}Shows "Using tool: web_search" indicator
observation{content: "- Python is..."}Shows collapsible "Observation" result

error{content: "All LLM providers failed"}Shows red error message in chat
scratchpad{content: "Thought:...\nAction:..."}Updates hidden debug panel (if open)
