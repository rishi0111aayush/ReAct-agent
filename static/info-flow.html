Main Information Flow
01
BROWSER — USER INPUT
static/index.html  |  .html  /  .js
.html .js text/plain base64

The user types a message in the browser. Optionally attaches an image. JavaScript reads the input as text/plain. Images are encoded as base64 data URLs (e.g. data:image/png;base64,...). Conversation history is held in memory as a JS array of objects.

Text String — max 4 000 chars, validated client-side
Image FileReader API → data:{mime};base64,{b64}
History JS Array<{user: string, assistant: string}>
02
HTTP REQUEST — POST /chat
Wire format over TCP/IP
application/json HTTP/1.1 cookie

JavaScript serialises all fields into a single JSON body and POSTs to /chat. The session cookie (set by Starlette SessionMiddleware) travels in the request header for auth identity.

{
  "message": string,
  "history": [{user, assistant}, …],
  "image": "data:image/…;base64,…" | null,
  "image_mime": "image/png" | null,
  "selected_model": "groq:llama-…" | null,
  "session_id": string | null
}
03
FASTAPI ROUTER
main.py  |  .py
.py Pydantic cookie/session HTTP 429

FastAPI auto-deserialises the JSON body into a Pydantic ChatRequest model (validated Python dataclass). SessionMiddleware decodes the signed cookie into a Python dict. Rate-limiting checks against Redis via cache.py. If allowed, calls the agent.

Input JSON body → Pydantic ChatRequest Python object
Auth Starlette signed cookie → request.session["user_id"]
Rate limit Redis key (IP or user_id) → count in-memory
Output Calls run_agent() → returns StreamingResponse
04
REACT AGENT LOOP
agent.py  |  .py
.py plain text Python dict

The core ReAct loop (max 5 iterations). On each iteration it builds a plain text prompt string, calls the LLM, parses the response with regex into a Python dict (thought, action, final_answer), and decides whether to call a tool or emit the final answer.

Scratchpad Plain text string — grows with each Thought/Observation
LLM output Raw string → regex parsed → Python dict
Yields Python generator — yields dicts → serialised to SSE
05
PROMPT TEMPLATE
prompts.py  |  .py
.py plain text f-string

SYSTEM_PROMPT and REACT_TEMPLATE are plain Python strings. The template is formatted with Python f-strings — injecting system instructions, conversation history, the user message, and the running scratchpad into one large plain-text prompt string.

Format Plain UTF-8 text — no markup or special encoding
Approx size ~len(prompt) / 4 tokens; capped at 8 192 ctx tokens
06
LLM API CALL
agent.py → external HTTPS endpoints
application/json text/event-stream HTTPS/TLS

The prompt is wrapped in an OpenAI-compatible JSON payload and sent via httpx. Groq streams tokens as Server-Sent Events (SSE) — each line is data: {"choices":[{"delta":{"content":"…"}}]}. Other providers return a single JSON response body.

Groq (stream)
→
Cerebras
→
Gemini
→
Ollama (local)
fallback chain
Request JSON — {model, messages, temperature, max_tokens, stream}
Groq resp SSE stream — data: {…delta…} lines + data: [DONE]
Others Single JSON — choices[0].message.content
Vision Image sent as {type:"image_url", image_url:{url:"data:…"}}
07
TOOL DISPATCHER
tools.py  |  .py
.py plain text regex parse

When the LLM emits Action: tool_name("arg"), the agent extracts it and calls dispatch_tool(). The action string is parsed by regex into tool_name + arg, then the matching Python function is called. Each tool returns a plain text string as its observation (except plot which returns a CHART:-prefixed JSON string).

Input Plain text — e.g. web_search("python async")
Parse Regex (\w+)\((.+)\) → tool name + arg string
Output Plain text observation, or CHART:{json} for plot
08
SSE EVENT STREAM
main.py → StreamingResponse  |  .py
text/event-stream application/json UTF-8

Each Python dict yielded by the agent is serialised with json.dumps() and wrapped in SSE format: data: {…}\n\n. The browser reads this stream via the EventSource API or fetch() with a readable stream reader. Response header: Cache-Control: no-cache.

Event type field values:

thought action observation answer_token done error chart iteration tokens timing model_used scratchpad
09
BROWSER — RENDER OUTPUT
static/index.html  |  .html  /  .js
.html .js DOM markdown

The browser's JS reads each SSE event. answer_token events are concatenated and rendered as Markdown → HTML. Chart events render a Chart.js canvas. Thought/action/observation events update the reasoning panel. Final state is a fully rendered HTML response in the DOM.

Text Markdown string → innerHTML (marked.js / DOMPurify)
Charts JSON spec → new Chart(canvas, config) (Chart.js)
Saved to DB JS calls POST /api/sessions/{id}/message with JSON body

Persistence & Cache Layer
💾
SQLite Database
boozo.db  |  .db  /  .sqlite3
.db SQL binary B-tree

Managed by db.py (.py). Three tables:

users — google_id, email, name, avatar
sessions — id, user_id, title, created_at
messages — session_id, role, content, ts

Wire format: Python sqlite3 module — parameterised SQL strings → binary SQLite file. Auto-title via simple heuristic on first user message.

⚡
Redis Cache
cache.py  |  .py  — in-memory
key-value TTL Redis protocol

Managed by cache.py (.py). Falls back to an in-process Python dict if Redis is unavailable.

search:{query} → text result · 1 h TTL
url:{url} → scraped text · 24 h TTL
mem:{user_id} → Redis list of fact strings · no TTL
rl:{key} → request count · 1 min window

📈
ChromaDB Vectors
chroma/ directory  — binary
float32 vectors HNSW index metadata JSON

Managed by tools.py (.py) via chromadb.PersistentClient. Stored in the chroma/ directory as binary files.

Text chunks → all-MiniLM-L6-v2 SentenceTransformer → 384-dim float32 embedding vectors → HNSW graph index. Metadata stored as JSON alongside each vector.

Auth Flow  (Google OAuth 2.0)
Handled entirely in main.py (.py) using authlib + Starlette sessions.
GET /auth/google Redirect 302 → Google OAuth consent URL (text/html)
callback params Query string: ?code=…&state=… → authlib token exchange → HTTPS/JSON
userinfo JSON from Google OIDC → Python dict → upserted into SQLite (boozo.db)
session google_id written to signed cookie — Starlette SessionMiddleware (itsdangerous)
GET /auth/me Cookie → google_id → SQLite lookup → JSON response
OIDC / JWT application/json signed cookie HTTPS/TLS .db SQLite
